01
Настройка
Выполняется один раз после установки Git
Без этих настроек коммиты будут анонимными и GitHub не свяжет их с твоим аккаунтом.
первый запуск
Имя и email
git config --global user.name "Твоё Имя" git config --global user.email "you@example.com"
GitHub будет показывать это имя под каждым твоим коммитом.
удобство
Редактор по умолчанию
git config --global core.editor nano
Используй nano вместо vim — проще для новичков. Или vim, если умеешь.
проверка
Посмотреть все настройки
git config --list
Выводит все текущие настройки Git — имя, email, редактор и т.д.
SSH ключ
Создать SSH ключ для GitHub
ssh-keygen -t ed25519 -C "you@example.com" cat ~/.ssh/id_ed25519.pub
Скопируй вывод → GitHub → Settings → SSH Keys. Больше не нужен пароль при push.
02
Старт проекта
Новый репозиторий или клонирование существующего
Новый проект с нуля
инициализация
Создать репозиторий в текущей папке
cd ~/projects/MyCppProject git init
Создаёт скрытую папку .git — это и есть весь репозиторий. Нет .git — нет Git.
Уже существующий проект с GitHub
клонирование
Скачать репозиторий с GitHub
# Через SSH (рекомендуется — не нужен пароль): git clone git@github.com:username/repo.git # Или через HTTPS: git clone https://github.com/username/repo.git
Создаёт папку с именем репозитория и скачивает все файлы + историю коммитов.
Связать локальный репо с GitHub
связать
Привязать GitHub к существующему локальному репо
git remote add origin git@github.com:username/repo.git git branch -M main git push -u origin main
Если создал git init локально, а потом создал пустой репо на GitHub — используй это чтобы их связать.

GitHub
статус
Что изменилось?
git status
Показывает какие файлы изменены, готовы к коммиту или не отслеживаются. Используй часто!
разница
Что именно изменилось?
# Изменения ещё не в staging: git diff # Изменения уже после git add: git diff --staged
Показывает построчно что добавлено (+) и удалено (-).
добавить
Подготовить файлы к коммиту (staging)
# Один файл: git add main.cpp # Несколько файлов: git add main.cpp utils.cpp # Все изменённые файлы в текущей папке: git add . # Все .cpp файлы: git add *.cpp
Git не записывает изменения автоматически. Нужно сначала «отметить» что войдёт в следующий коммит.
коммит
Сохранить snapshot
# Обычный коммит: git commit -m "Добавил класс Vector3D" # Добавить и сразу закоммитить (только уже отслеживаемые файлы): git commit -am "Исправил баг в рендере"
Коммит — это сохранение с сообщением. Пиши понятно: что и зачем сделал. Не «фикс», а «Исправил краш при нулевом указателе в Renderer».
04
Ветки (branches)
Параллельные версии проекта
Ветка = параллельная версия проекта. Работай на отдельной ветке — main всегда остаётся рабочим.
просмотр
Список веток
# Локальные: git branch # Все включая удалённые: git branch -a
Текущая ветка помечена звёздочкой *.
создать
Создать и переключиться
# Создать и сразу переключиться: git checkout -b feature/physics-engine # Современный синтаксис (Git 2.23+): git switch -c feature/physics-engine
Называй ветки понятно: feature/название, fix/баг, refactor/что.
слияние
Влить ветку в main
git checkout main git merge feature/physics-engine
Переходишь на main, затем вливаешь в него изменения из feature-ветки.
удалить
Удалить ветку
# После merge (безопасно): git branch -d feature/physics-engine # Принудительно: git branch -D feature/physics-engine # Удалить на GitHub: git push origin --delete feature/physics-engine
конфликты
Если возник конфликт при merge
<<<<<<< HEAD твой код здесь ======= чужой код здесь >>>>>>> feature/physics-engine
# После ручного исправления файла: git add main.cpp git commit -m "Разрешил конфликт в main.cpp"
Конфликт = два человека изменили одно место. Git не знает чья правка верная — решаешь ты. Открой файл, найди маркеры <<<<<<<, выбери нужный код, сохрани.
05
GitHub / Remote
Работа с удалёнными репозиториями
origin — это просто псевдоним для адреса твоего репозитория на GitHub. Можно назвать как угодно, но origin — стандарт.
push
Отправить коммиты на GitHub
# Обычный push: git push # Первый push новой ветки: git push -u origin feature/physics-engine # После -u достаточно просто: git push
Флаг -u запоминает куда пушить — больше не нужно указывать origin и ветку.
pull
Получить изменения с GitHub
# Скачать И влить в текущую ветку: git pull # Только скачать (без слияния): git fetch origin
pull = fetch + merge. Делай pull перед началом работы каждый день.
просмотр
Просмотр удалённых репо
git remote -v
Показывает адреса fetch и push. Полезно проверить правильный ли URL.
pull request
Workflow с Pull Request
git checkout -b feature/new-shader # делаешь изменения... git add . git commit -m "Добавил PBR шейдер" git push -u origin feature/new-shader # → GitHub → New Pull Request
Pull Request — предложение влить ветку в main. Другие могут проверить код перед merge.
06
История коммитов
Просмотр, поиск, навигация
лог
Просмотр истории
# Полный лог: git log # Компактный (одна строка): git log --oneline # Красивый граф с ветками: git log --oneline --graph --all # Последние 5 коммитов: git log -5 --oneline
поиск
Найти коммит
# По сообщению: git log --oneline --grep="шейдер" # По автору: git log --oneline --author="Vasya" # Когда добавилась функция: git log -S "renderShadow" --oneline
просмотр
Посмотреть конкретный коммит
# Что изменилось в коммите: git show abc1234 # Конкретный файл в том коммите: git show abc1234:src/main.cpp
файл
История конкретного файла
git log --oneline -- main.cpp # Кто изменял каждую строку: git blame main.cpp
07
Отмена изменений
Исправляем ошибки
Правило: до push — можно переписывать историю. После push — только новые коммиты (revert).
staging
Убрать файл из git add
git restore --staged main.cpp
Файл останется изменённым, но не войдёт в следующий коммит.
откат файла
Откатить файл к последнему коммиту
git restore main.cpp
Осторожно! Все несохранённые изменения в этом файле пропадут безвозвратно.
amend
Исправить последний коммит
# Только сообщение: git commit --amend -m "Правильное сообщение" # Добавить забытый файл: git add забытый.cpp git commit --amend --no-edit
Только если ещё не запушил! После push используй revert.
revert
Отменить коммит (безопасно)
git revert abc1234
Создаёт новый коммит, который отменяет изменения. История не портится. Можно пушить. Это правильный способ после push.
reset
Сбросить к коммиту (осторожно!)
# Мягкий: коммиты убраны, изменения в staging: git reset --soft HEAD~1 # Средний: коммиты убраны, изменения в рабочей папке: git reset --mixed HEAD~2 # Жёсткий: всё удалено без возврата: git reset --hard HEAD~1
HEAD~1 = один коммит назад. HEAD~2 = два коммита назад. Только до push!
08
Продвинутые команды
Stash, rebase, cherry-pick, tags
stash
Спрятать незаконченную работу
# Спрятать: git stash # Спрятать с именем: git stash push -m "WIP: рефакторинг" # Список: git stash list # Достать последнее: git stash pop
Нужно срочно переключиться, но коммит делать рано? Прячь в stash.
rebase
Перенести ветку на другую основу
# Перебазировать на текущий main: git checkout feature/physics git rebase main # Интерактивный (переписать 3 коммита): git rebase -i HEAD~3
Делает историю линейной. Только для локальных веток или если работаешь один.
cherry-pick
Взять один коммит из другой ветки
git cherry-pick abc1234
Полезно если нужно срочно перенести одно исправление в main, не сливая всю ветку.
tags
Пометить версию тегом
git tag -a v1.0.0 -m "Первый релиз" git push origin v1.0.0 git push origin --tags
Теги — именованные точки в истории. Используй для версий релизов.
C++
C++ и CLion
Настройка .gitignore и правильный workflow
CLion использует CMake. Важно создать .gitignore до первого git add . — иначе гигабайты бинарников попадут в репо.
Что коммитить, а что нет
✓ Коммитить
src/*.cpp src/*.h include/*.h CMakeLists.txt README.md .gitignore tests/*.cpp
✗ Не коммитить
cmake-build-debug/ cmake-build-release/ .idea/workspace.xml *.o *.exe *.out build/ out/
Файл .gitignore для CLion / C++
.gitignore
# CMake / Build артефакты cmake-build-debug/ cmake-build-release/ cmake-build-*/ build/ out/ # CLion IDE файлы .idea/ # Скомпилированные файлы *.o *.obj *.a *.lib *.so *.dll *.exe *.out # Системные файлы .DS_Store Thumbs.db
старт проекта
Типичный старт C++ проекта с нуля
# 1. Создаёшь проект в CLion cd ~/CLionProjects/MyProject # 2. Инициализируешь Git git init # 3. Создаёшь .gitignore (скопируй блок выше) # 4. Первый коммит git add . git commit -m "Начальная структура проекта" # 5. Создаёшь пустой репо на github.com (без README!) # 6. Связываешь и пушишь git remote add origin git@github.com:ты/проект.git git branch -M main git push -u origin main
если .idea попало
Убрать .idea из отслеживания
git rm -r --cached .idea/ echo ".idea/" >> .gitignore git add .gitignore git commit -m "Убрал .idea из репо" git push
Флаг --cached убирает из Git, но не удаляет файлы с диска.
workflow
Ежедневный workflow
# Начало дня: git pull # Новая фича: git checkout -b feature/collision # Работаешь в CLion... git add src/Collision.cpp src/Collision.h git commit -m "Добавил AABB коллизии" git push -u origin feature/collision
